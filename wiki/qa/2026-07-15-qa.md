# Q&A — 2026-07-15 (Iteration 40)

## Q1: What is the complete attack taxonomy for the agent skill supply chain?

Based on today's ingestion of MalTool, BadSkill, and SkillJect (plus prior ClawHavoc), the attack taxonomy now spans four distinct threat surfaces:

| Attack | Threat Surface | Mechanism | Defense |
|--------|---------------|-----------|---------|
| **MalTool** (arXiv:2602.12194) | Tool code implementation | Coding LLM generates malicious tool code (standalone or embedded) | SkillSpector (YARA + semantic scanning) |
| **BadSkill** (arXiv:2604.09378) | Model weights in skills | Backdoor-fine-tuned model embedded in skill; semantic compositional triggers | Behavioral vetting, model provenance (SkillFortify) |
| **SkillJect** (arXiv:2602.14211) | SKILL.md + helper scripts | Two-channel: payload in helper script + front-loaded inducement in SKILL.md | SkillSpector (scan helper scripts), SkillGuard (runtime) |
| **ClawHavoc** (prior) | Marketplace distribution | ~1,184 malicious skills uploaded to ClawHub; dual-vector (prompt injection + shell) | Marketplace scanning (SkillSieve, agentskill.sh) |

Each attack targets a different layer of the skill bundle (code, model, instructions, distribution) and requires a different defense mechanism. No single defense covers all four.

## Q2: Does the defense stack now achieve full lifecycle coverage?

Yes, as of today's ingestion:

| Lifecycle Stage | Defense Framework | Coverage |
|----------------|-------------------|----------|
| Pre-installation scanning | SkillSpector (NVIDIA) | YARA + LLM semantic + OWASP/MITRE |
| Supply chain verification | SkillFortify | Formal (Dolev-Yao, abstract interpretation, lockfile) |
| Runtime permission | SkillGuard | Dual-plane (context + action), deny-by-default |
| MCP server access control | AgentBound | Declarative policy (Android model), enforcement engine |
| Marketplace detection | SkillSieve | Three-layer triage (heuristic→LLM→jury) |
| Publication governance | NVIDIA-Verified | Catalog → scan → sign → document → distribute |

The gap: **model-in-skill verification** (BadSkill's threat surface) is not yet covered by any defense framework. SkillSpector scans code, SkillFortify verifies supply chain, SkillGuard governs runtime — but none verifies that model weights embedded in skills are not backdoor-fine-tuned.

## Q3: What does SWE-Skills-Bench tell us about skill utility?

The finding is sobering: **80% of SWE skills (39/49) yield zero pass-rate improvement**, and the average gain is only +1.2%. Three skills actually degrade performance (up to -10%) due to version-mismatched guidance. Token overhead can reach +451% with no benefit.

This means:
- Skills are a **narrow intervention**, not a universal booster
- Utility depends on **domain fit, abstraction level, and contextual compatibility**
- The field needs **rigorous evaluation** before adoption, not just enthusiasm
- The **less-is-more effect** (SkillReducer) is validated: many skills are actively wasteful

Combined with SkillReducer's finding (55K skills studied, 60%+ non-actionable), the evidence increasingly suggests that skill quality matters far more than skill quantity.

## Q4: How does the Externalization framework connect to existing wiki concepts?

The Externalization paper provides the **theoretical foundation** for the entire skill bundles knowledge base:

- **Skills = externalized procedural expertise** — formal definition of what skills are
- **Harness engineering = governed execution** — connects to SkillGuard, SkillFortify, AgentBound
- **Self-evolving harnesses** — connects to SkillClaw, CoEvoSkills, Skill1
- **Shared agent infrastructure** — connects to SkillClaw's shared repository
- **Parametric/externalized trade-off** — explains why SkillReducer's less-is-more works (some capabilities are better left parametric)
- **Weights→Context→Harness progression** — historical framework for understanding the field's evolution

The four-form taxonomy (memory, skills, protocols, harness) provides a unified classification system that the wiki can use to organize its examples.

## Q5: What are the gaps identified by today's research?

1. **Model-in-skill verification gap** — No defense framework addresses BadSkill's threat surface (backdoor models in skills)
2. **Skill utility evaluation gap** — SWE-Skills-Bench covers SWE; no equivalent for other domains (research, creative, data science)
3. **Skill versioning gap** — SWE-Skills-Bench's version-mismatch finding shows skills need explicit version compatibility declarations
4. **MCP security beyond AgentBound** — AgentBound covers access control; what about MCP server code verification, MCP protocol-level attacks?
5. **MalTool defense gap** — MalTool explicitly shows existing detection methods are insufficient; new approaches needed
6. **Collective evolution governance** — SkillClaw enables collective evolution, but who governs what gets propagated? (security implications of shared skill repositories)
