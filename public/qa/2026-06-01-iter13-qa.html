Q1: Which bundles combine skill-level validation (structure + content quality) with registry-level security scanning?

Search query: skill bundles with both skill-validator type checks AND 12-category threat scanning

Answer:

Bundle Skill-Level Validation Registry Security Scanning
agentskill sh ags security scoring review-skill (10 quality dimensions) 12-category server-side scanner
skill validator cli Full lifecycle (structure, content, contamination, link rot, LLM-as-judge) None — local-only
validate skill github action Structural spec compliance None — CI/CD only

Three-layer coverage matrix (NEW):

  • Pre-commit layer: skill-validator (Go binary, 13 platform hooks)
  • CI/CD layer: Validate Skill Action (GitHub Action, JSON outputs)
  • Registry layer: agentskill.sh (12-category threat scan, 100k+ skills)

Each layer catches what the others miss:

  • skill-validator catches deep quality issues (contamination, LLM-as-judge) but is opt-in
  • Validate Skill catches structural regressions on every PR but doesn't run LLM
  • ags catches registry-level threats (malicious payloads, supply chain) but only at install

Cross-reference: three layer validation stack — the architectural pattern

Q2: What ontology patterns appear most frequently alongside agent skills?

Search query: ontology + agent skill combinations

Answer (top 6):

  1. OBO Foundry ontologies (GO, Mondo, CL, Uberon, EFO, HPO) — 4 production users of ai4curation curation skills
  2. SKOS vocabularies — implicit in many taxonomic skill bundles
  3. SHACL shape libraries — graphguard os guardrails, xpshacl explainable shacl, schimatos shacl knowledge graph, text2shacl multi agent shacl
  4. OWL ontologies (general) — 8+ examples in catalog
  5. MITRE ATT&CK / NIST — claude skills ecosystem 1116 (754 cybersecurity skills)
  6. Custom domain ontologies — onto llm mapping for SSSOM mapping generation

Convergence: Ontology-as-context is the second most common context element type (after SHACL validation). The pattern of "skill operates on formal ontology terms" is the key bundle shape.

Q3: How do successful bundles handle rule integration?

Search query: how skills combine with formal rules

Answer (5 patterns):

  1. External rule engine (e.g., veto agent authorization — authorization kernel separate from skills)
  2. Embedded rules in skill body (e.g., superpowers engineering discipline skills — Iron Laws / Red Flags in the skill text)
  3. Rule-as-SHACL (graphguard os guardrails — guardrails expressed as SHACL shapes)
  4. Rule-as-LLM-as-judge (skill validator cli — LLM evaluates quality dimensions)
  5. Threat taxonomy as rule schema (agentskill sh ags security scoring — 12-category threat model defines what to scan for)

Convergence: There is no single "right" way — it depends on whether rules are static (SHACL) or dynamic (LLM-judge), and whether they're declarative (Veto) or embedded (Superpowers). Five distinct patterns in the catalog.

Q4: What is the bundle density leader and what makes it dense?

Top 5 by bundle density (skills × context elements × distribution channels):

  1. agentskill sh ags security scoring — 3 skills + 12-category threat taxonomy + 10-dim quality scoring + content-SHA versioning + 4 distribution channels (npm, plugin marketplace, slash command, JSON) + telemetry (auto-rate, 110k scans) = ~30 distinct elements
  2. ai4curation curation skills — 7 skills + 5 ontologies + OAK/ROBOT/DOSDP vocab + 4 production users + Justfile reproducibility = ~17 elements
  3. skill validator cli — 1 tool with 7 commands + 13 platform hooks + LLM-as-judge + contamination detection + 4 content metrics = ~25 elements
  4. text2shacl multi agent shacl — 4-stage pipeline + 5 LLMs + 2 input versions + 2 merge strategies + 3 storage backends (Chroma, Redis, ontology) = ~16 elements
  5. validate skill github action — 1 action + 5 inputs + 3 outputs + 12 validation rules = ~21 elements

Pattern: Density leaders combine multiple types of context element (validation + scoring + provenance) with multiple distribution channels (CLI + marketplace + CI + plugin). Single-purpose tools (validate-skill Action) are less dense but more focused.

Q5: What gaps remain in the catalog?

Top 5 open gaps (from analysis):

  1. Bundle-level validation — All three validation layers target individual skills; bundle-level interaction effects (privilege accumulation, version skew) are an open problem. No example yet.
  2. SLSA / in-toto attestation — The three validation layers produce independent signals; no provenance chain linking them.
  3. A2A (Agent-to-Agent) protocol — Listed in last week's priorities; no example found yet.
  4. SHACL round-trip evaluation — text → shapes → text fidelity not yet empirically tested.
  5. Skill provenance beyond OpenSSF signing + content-SHA — In-toto, sigstore, TUF not yet represented.

Conclusion

Iteration 13 is the most pattern-rich iteration to date — the three-layer validation stack + bidirectional SHACL bridge are the two architectural patterns that the catalog will likely build on for the next several iterations. The next research priority should be the gaps identified in Q5, especially bundle-level validation and SLSA-style attestation.
